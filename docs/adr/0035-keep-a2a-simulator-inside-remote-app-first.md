# Keep A2A Simulator Inside Remote App First

The A2A Simulator should initially live inside `apps/remote` rather than a shared package, with scenario YAML files under `apps/remote/scenarios/`. The initial scenario set contains one happy-path scenario that shows as many useful A2A states and stream event types as possible while still remaining a coherent successful flow. That happy path includes one recoverable `auth-required` pause, such as approval before pushing a branch or opening a pull request. For the first pass, any subsequent user input to the same task is treated as approval, then the task resumes to `working` and completes. The simulator is still being shaped by the Jira remote-agent sample, so keeping it local reduces premature API design. It can be extracted once the API is proven and a second consumer exists.

Post-hoc implementation note: the scenario directory has grown beyond the initial happy path into executable examples for authorization, clarification, rejection, failure, cancellation, and resubscribe behavior. That does not contradict keeping the simulator local first; the targeted scenarios are the acceptance corpus that is proving the simulator API before extraction.
