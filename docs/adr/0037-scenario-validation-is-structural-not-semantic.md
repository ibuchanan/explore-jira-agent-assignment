# Scenario Validation Is Structural Not Semantic

The A2A Simulator validates scenarios enough to catch malformed YAML, missing required structural fields, and obvious typos. Malformed scenario structure that prevents execution uses `400` Problem Details. The simulator does not block scenarios solely for A2A or Jira semantic non-compliance because scenarios may intentionally model unusual or invalid A2A streams for client testing. Instead, it logs non-blocking `422` Compliance Warnings when a structurally valid scenario appears non-compliant with A2A or Jira remote-agent expectations. Compliance Warnings use the project's Problem Details fields (`type`, `title`, `status`, `detail`, `timestamp`, and optional `instance`) but are logs, not response errors. Their `type` values should point to public specification URLs where possible, using coarse-grained URLs for broad warnings and specific anchors for fine-grained warnings.
